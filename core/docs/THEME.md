# Crawl Theme System

The Crawl theme system generates, persists, and applies colour palettes across the desktop environment. It lives primarily in the `crawl-theme` crate with IPC types in `crawl-ipc` and orchestration in `crawl-sysd`.

## Architecture

```
                    CLI (crawl theme --set mocha)
                           │
                     IPC command
                           │
                           ▼
              ┌──────────────────────────────────┐
              │    crawl-sysd                    │
              │  ThemeService                    │
              │   ├── ThemeConfig                 │
              │   │   ├── variant                │
              │   │   ├── accent                 │
              │   │   ├── cursor                 │
              │   │   └── gtk                    │
              │   ├── resolve_and_apply          │
              │   ├── ThemeGenerateFromPredefined│
              │   └── ThemeGenerateTerminal      │
              └─────────┬────────────────────┬──┘
                        │                    │
          ┌─────────────┼─────────────┐      │
          ▼             ▼             ▼      ▼
    crawl-theme    apply/         IPC         IPC
    ┌──────────┐  ┌──────────┐  broadcast  → TemplateProcessor.qml
    │ Color    │  │ GTK      │  ┌──────────┐  ┌────────────────────┐
    │ Palette  │  │ Qt       │  │ ThemeCh. │  │ Terminal gen       │
    │ Dynamic  │  │ Cursor   │  │ Event    │  │ Scheme expansion   │
    │ Scheme   │  │ Niri     │  │          │  │ (was Python,       │
    │ Variants │  └──────────┘  │  → QML   │  │  now Rust daemon)  │
    │ Theme    │                └──────────┘  └────────────────────┘
    │ Engines  │
    │ Terminal │
    └──────────┘
```

Three layers:

| Layer | Crate | Responsibility |
|-------|-------|----------------|
| **Core engine** | `crawl-theme` | `Color`, `Palette`, `DynamicScheme`, predefined variants, theme engines (wallust-style), predefined scheme expansion, terminal theme generation, system apply (GTK/Qt/Cursor/Niri) |
| **Daemon** | `crawl-sysd` | `ThemeService` — loads/saves config, resolves palettes, orchestrates apply, handles IPC for all theme commands including generation and terminal output |
| **IPC types** | `crawl-ipc` | Commands (`ThemeGet`, `ThemeSet`, `ThemeGenerate`, `ThemeGenerateFromImage`, `ThemeGenerateFromPredefined`, `ThemeGenerateTerminal`) |
| **CLI** | `crawl-cli` | `crawl theme --get/--list/--set/--generate/--image` |

### Dual-path Architecture

Theme processing follows two independent paths coordinated by `TemplateProcessor.qml`:

| Path | Input | Data Flow |
|------|-------|-----------|
| **Wallpaper** | Image file → `extract.rs` dominant color → `generate_theme()` → `template-processor.py` renders templates |
| **Predefined scheme** | 14-color JSON (`cPrimary`, etc.) → daemon IPC `ThemeGenerateFromPredefined` → 48-key expanded palette → `template-processor.py` renders templates |

Terminal configs in both paths are now generated by the Rust daemon (`ThemeGenerateTerminal` IPC) instead of Python.

## Colour Model

### Color (`crawl-theme/src/color.rs`)

```rust
pub struct Color(pub u8, pub u8, pub u8);  // RGB
```

- Parsed from hex strings: `Color::from_hex("#cba6f7")` (supports `#rgb` and `#rrggbb`)
- Serialised as `#rrggbb` format via serde
- `luminance()` — relative luminance (WCAG) for determining text contrast

### HCT (`crawl-theme/src/dynamic/hct/`)

HCT (Hue-Chroma-Tone) is the colour space used by Material Design 3 for dynamic colour generation. Implemented in pure Rust:

| Component | File | Description |
|-----------|------|-------------|
| `Hct` | `hct.rs` | Hue (0-360), Chroma (0-~180), Tone (0-100), ARGB conversion |
| `Cam16` | `cam16.rs` | Simplified CAM16 colour appearance model (hue, chroma) |
| `TonalPalette` | `tonal.rs` | Fixed hue+chroma, tone-indexed cache (returns `Color` for tone 0-100) |
| `lab.rs` | — | CIELAB D65 conversions for colour distance |

Tonal palettes are generated from a seed colour:

```
Seed → HCT → 5 TonalPalettes (primary, secondary, tertiary, neutral, neutral_variant)
         ↓
    Each palette has 101 tones (0-100)
         ↓
    Tone constants map tones to Catppuccin palette fields
```

### Colour Utilities (`crawl-theme/src/color.rs` and `contrast.rs`)

```rust
// Color struct
pub struct Color(pub u8, pub u8, pub u8);

// HSL conversions
fn to_hsl(&self) -> (f32, f32, f32)  // (hue 0-360, saturation 0-1, lightness 0-1)
fn from_hsl(h: f32, s: f32, l: f32) -> Color

// Colour manipulation
fn adjust_lightness(&self, amount: f32) -> Color
fn shift_hue(&self, degrees: f32) -> Color
fn saturate(&self, factor: f32) -> Color
fn adjust_surface(&self, sat_cap: f32, lightness: f32) -> Color
fn hue_distance(h1: f32, h2: f32) -> f32

// Contrast (WCAG)
fn relative_luminance(&self) -> f32
fn contrast_ratio(a: &Color, b: &Color) -> f32
fn ensure_contrast(target: &Color, bg: &Color, min_ratio: f32, lock_direction: Option<bool>) -> Color
```

`ensure_contrast` performs a binary search on lightness to guarantee the target meets the minimum contrast ratio against the background. `lock_direction` forces lightening (`Some(true)`) or darkening (`Some(false)`); `None` auto-detects.

### Image Extraction (`crawl-theme/src/dynamic/extract.rs`)

`dominant_color(path)` extracts a seed colour from a wallpaper image:

1. Resize to 112×112 pixels
2. Quantise to 128 colours via Xiaolin Wu algorithm
3. Refine with weighted k-means in Lab space
4. Score colours by chroma + pixel proportion (Material Score)
5. Return highest-scored colour

## Palette

The `Palette` struct (`crawl-theme/src/palette.rs`) holds 26 named colours following the Catppuccin naming scheme:

```rust
pub struct Palette {
    // Surfaces
    pub base: Color,       // Dark: #1e1e2e    Light: #eff1f5
    pub mantle: Color,     // Dark: #181825    Light: #e6e9ef
    pub crust: Color,      // Dark: #11111b    Light: #dce0e8

    pub surface0: Color,   // Dark: #313244    Light: #ccd0da
    pub surface1: Color,   // Dark: #45475a    Light: #bcc0cc
    pub surface2: Color,   // Dark: #585b70    Light: #acb0be

    pub overlay0: Color,
    pub overlay1: Color,
    pub overlay2: Color,

    // Text
    pub text: Color,
    pub subtext0: Color,
    pub subtext1: Color,

    // 14 accent colours
    pub rosewater: Color,
    pub flamingo: Color,
    pub pink: Color,
    pub mauve: Color,
    pub red: Color,
    pub maroon: Color,
    pub peach: Color,
    pub yellow: Color,
    pub green: Color,
    pub teal: Color,
    pub sky: Color,
    pub sapphire: Color,
    pub blue: Color,
    pub lavender: Color,

    // Extension point for user-defined colours
    pub extra: HashMap<String, Color>,
}
```

Methods:
- `accent(name) -> Option<&Color>` — lookup accent by name string
- `to_map() -> HashMap<String, String>` — flat hex map (used for IPC broadcast)

### Dynamic

`ThemeVariant::Dynamic { seed, mode, variant }` — generates a palette from a seed colour using Material Design 3 tonal palettes mapped to Catppuccin-shaped slots. The seed hex is persisted in `theme.toml` so the palette is reproducible without re-extraction.

### Custom

`ThemeVariant::Custom(path_or_name)` — tries to load a TOML palette file from the given path. If the path doesn't exist, falls back to Catppuccin Mocha. See [Custom Palettes](#custom-palettes) for the TOML format.

### Light/Dark mode

```rust
impl ThemeVariant {
    pub fn is_dark(&self) -> bool {
        // Light variants: Latte, GruvboxLight, KanagawaLight, RosePineLight
        // Dynamic with mode == "light"
        // All others (including Custom) default to dark
    }
}
```

## Dynamic Scheme Types

When generating a dynamic theme, the `SchemeType` controls how hue/chroma are distributed across the five tonal palettes:

| Scheme | Description |
|--------|-------------|
| `TonalSpot` | Default. Seed hue for primary/tertiary, neutral-cool for others. |
| `Content` | Derives primary/ secondary from seed content, uses neutral tones. |
| `Monochrome` | Zero chroma on all palettes — pure greyscale. |
| `Vibrant` | High chroma on primary/secondary/tertiary. |
| `Faithful` | Primary matches seed as closely as possible. |
| `FruitSalad` | Wide hue variation between palettes. |
| `Rainbow` | Even hue spacing across palettes (360° / 5). |
| `Muted` | Low chroma on all palettes — subdued. |

Each scheme type defines `PaletteParams { primary, secondary, tertiary, neutral, neutral_variant }` — `(hue_offset, chroma_multiplier)` pairs.

## Theme Engines

`crawl-theme::dynamic::generate` provides four theme engine pairs (dark/light) that convert a 5-color palette extracted from a wallpaper into a full 48-key MD3 palette:

| Engine | Function | Used by Scheme Types | Description |
|--------|----------|---------------------|-------------|
| **Normal Dark** | `generate_normal_dark()` | vibrant, faithful, dysfunctional | High saturation, high contrast, dark surface tones |
| **Normal Light** | `generate_normal_light()` | vibrant, faithful, dysfunctional | High saturation, light surface, dark text |
| **Muted Dark** | `generate_muted_dark()` | muted | Low saturation (`≤15%`), subdued palette |
| **Muted Light** | `generate_muted_light()` | muted | Low saturation, light surface |

The master dispatcher `generate_theme(palette, mode, scheme_type)` routes to the correct engine:

```rust
pub fn generate_theme(palette: &[Color], mode: &str, scheme_type: &str) -> HashMap<String, String> {
    match scheme_type {
        "vibrant" | "faithful" | "dysfunctional" => { /* generate_normal_dark/light */ }
        "muted" => { /* generate_muted_dark/light */ }
        _ => { /* Material Design 3 DynamicScheme */ }
    }
}
```

All engines build the palette via `collect_theme_map()` which accepts 48 color parameters and returns a `HashMap<String, String>` with these keys:

`primary`, `on_primary`, `primary_container`, `on_primary_container`, `primary_fixed`, `primary_fixed_dim`, `on_primary_fixed`, `on_primary_fixed_variant`, `secondary/*`, `tertiary/*`, `error/*`, `surface`, `on_surface`, `surface_variant`, `on_surface_variant`, `surface_dim`, `surface_bright`, `surface_container_lowest`, `surface_container_low`, `surface_container`, `surface_container_high`, `surface_container_highest`, `outline`, `outline_variant`, `shadow`, `scrim`, `inverse_surface`, `inverse_on_surface`, `inverse_primary`, `background`, `on_background`, `surface_tint`.

## Predefined Scheme Expansion

`expand_predefined_scheme(scheme_data, mode)` in `crawl-theme::dynamic::generate` converts a 14-color predefined scheme into the full 48-key MD3 palette.

### Input format

```json
{
  "cPrimary": "#FFC864",
  "cOnPrimary": "#1A1A1A",
  "cSecondary": "#64C8FF",
  "cOnSecondary": "#1A1A1A",
  "cTertiary": "#C864FF",
  "cOnTertiary": "#1A1A1A",
  "cError": "#FF4444",
  "cOnError": "#FFFFFF",
  "cSurface": "#1E1E2E",
  "cOnSurface": "#CDD6F4",
  "cSurfaceVariant": "#45475A",
  "cOnSurfaceVariant": "#BAC2DE",
  "cOutline": "#585B70",
  "cShadow": "#000000"  // optional, falls back to cSurface
}
```

### Expansion logic

1. **Container colors** — derived via `make_container_dark/light()` which adjusts lightness and saturation
2. **"On container" colors** — contrast-checked via `ensure_contrast()` with binary search on lightness
3. **Fixed colors** — bright/dim variants via `make_fixed_dark/light()`
4. **Surface container levels** — interpolated between `cSurface` and `cSurfaceVariant`
5. **Inverse colors** — swapped lightness for dark/light mode
6. **Outline variant** — derived from outline by adjusting lightness

This runs in the daemon via the `ThemeGenerateFromPredefined` IPC command, replacing the old Python `expand_predefined_scheme`.

## Terminal Theme Generation

`crawl-theme::apply::terminal` generates terminal emulator colour configuration files from colour scheme data.

### Pipeline

```
Scheme JSON (14-color + terminal section)
        │
        ▼
TerminalColors::from_dict(data, scheme_colors)
  - Extracts foreground, background, cursor, selection
  - Parses normal/bright 8-color arrays
  - Derives tab bar colours, borders, indexed colors from UI scheme
        │
        ▼
TerminalGenerator::new(terminal_colors)
  - generate_foot()    → INI format
  - generate_ghostty() → key=value format
  - generate_kitty()   → key value format
  - generate_alacritty() → TOML format
  - generate_wezterm() → TOML with tab bar + indexed colors
        │
        ▼
write_terminal_themes(scheme_data, outputs)
  - Writes each terminal config to its output path
  - Creates parent directories as needed
```

The daemon exposes this via `ThemeGenerateTerminal` IPC. `TemplateProcessor.qml` calls this to generate terminal configs for enabled terminals, replacing the old Python subprocess.

### Supported terminals

| Terminal | File Format | Output Example |
|----------|-------------|----------------|
| foot | INI | `regular0=1e1e2e` / `bright0=...` |
| ghostty | key=value | `palette = 0=#1e1e2e` |
| kitty | key value | `color0 #1e1e2e` |
| alacritty | TOML | `[colors.normal]` `black = '#1e1e2e'` |
| wezterm | TOML | `[colors]` `ansi = [...]` + `[colors.tab_bar]` |

## IPC Protocol

### Commands (client → daemon)

All defined as `CrawlCommand` variants in `crawl-ipc/src/commands.rs`:

| Command | Parameters | Response |
|---------|-----------|----------|
| `ThemeGet` | — | `ThemeChangedEvent` JSON |
| `ThemeList` | — | `["mocha","latte",..."rose-pine-light"]` |
| `ThemeSet` | `{ name, accent? }` | Updated `ThemeChangedEvent` |
| `ThemeGenerate` | `{ color, scheme?, mode? }` | Updated `ThemeChangedEvent` |
| `ThemeGenerateFromImage` | `{ path, scheme?, mode? }` | Updated `ThemeChangedEvent` |
| `ThemeGenerateFromPredefined` | `{ scheme_json, mode }` | 48-key MD3 palette map `{ primary, on_primary, ... }` |
| `ThemeGenerateTerminal` | `{ scheme_json, outputs_json }` | `{ ok: true }` or error |

### Events (daemon → client)

`ThemeEvent` variants in `crawl-ipc/src/events.rs`:

| Event | Data | Trigger |
|-------|------|---------|
| `Theme::Changed` | `{ name, source, scheme }` | Theme variant/accent changed |
| `Theme::Applied` | `{ app, path }` | Theme applied to a system target |
| `Theme::Generated` | `{ name, color, path }` | New theme generated from colour/image |

### Response payload

`ThemeChangedEvent` (`crawl-theme/src/ipc.rs`):

```json
{
  "variant": "mocha",
  "accent": "#cba6f7",
  "palette": {
    "base": "#1e1e2e",
    "mantle": "#181825",
    "crust": "#11111b",
    "surface0": "#313244",
    "text": "#cdd6f4",
    "mauve": "#cba6f7",
    "accent": "#cba6f7",
    ...
  }
}
```

The QML frontend receives this via `IPCService.qml` and writes it to `colors.json`.

## Apply Targets

All in `crawl-theme/src/apply/`:

### GTK (`gtk.rs`)

Writes to `~/.config/gtk-3.0/` and `~/.config/gtk-4.0/`:

- **`settings.ini`**: `gtk-application-prefer-dark-mode`, `gtk-cursor-theme-name`, `gtk-cursor-theme-size`, `gtk-font-name`, `gtk-icon-theme-name`
- **`gtk.css`**: Colours for `@define-color theme_bg_color`, `theme_fg_color`, `theme_selected_bg_color`, `theme_base_color` via palette

Optional D-Bus signal via `zbus` (requires `gtk` feature) to trigger live reload via `NameOwnerChanged`.

### Qt (`qt.rs`)

Writes to `~/.config/qt5ct/` and `~/.config/qt6ct/`:

- **`colors.conf`**: Qt colour roles mapped from palette: `base`, `text`, `selection_bg` (uses resolved accent), `window`, `button`, `link`
- **`qt5ct.conf` / `qt6ct.conf`**: Sets `color_scheme_path` and `style=kvantum`

### Cursor (`cursor.rs`)

Writes `~/.icons/default/index.theme`:

```ini
[Icon Theme]
Inherits=catppuccin-mocha-dark-cursors
```

### Niri (`niri.rs`)

Writes to `~/.config/niri/theme.kdl` (Linux-only):

```kdl
layout {
    focus-ring { active-color "#cba6f7"; }
    border { active-color "#cba6f7"; }
}
ui {
    border { active-color "#cba6f7"; focused-color "#cba6f7"; }
    window-marker { active-color "#cba6f7"; }
}
background-color #1e1e2e
default-window-background-color #313244
```

User must add `include "theme.kdl";` to their `config.kdl`.

Sends `niri msg action reload-config` after writing.

### Sequence

`ThemeService::resolve_and_apply()` calls all four in order: GTK → Qt → Cursor → Niri. If any fails, a warning is logged but the remaining applies continue.

## CLI Usage

```bash
# Show current theme

crawl theme --list
```

## QML Integration

The QML frontend receives theme changes via the daemon's IPC broadcast:

1. `ThemeService::resolve_and_apply()` calls `ThemeChangedEvent::from_config()` returning the palette map
2. The daemon publishes `CrawlEvent::Theme(ThemeEvent::Changed { ... })` on the event bus
3. `IPCService.qml` receives it and calls `ColorSchemeService::applyScheme()`
4. `ColorSchemeService.qml` writes `colors.json` with MD3-mapped colours
5. `Color.qml` singleton watches `colors.json` and animates 16 colour properties (`cPrimary`, `cOnPrimary`, `cSurface`, etc.)

The bridge between Catppuccin naming (26 colours) and MD3 naming (16 colours) happens in `ColorSchemeService.qml::writeColorsToDisk()`.

### TemplateProcessor.qml (Dual-path dispatcher)

`TemplateProcessor.qml` orchestrates both wallpaper-based and predefined-scheme theme generation:

| Path | Trigger | Data flow |
|------|---------|-----------|
| **Wallpaper** | `processWallpaperColors(path, mode)` | → `executeWallpaperColors()` → writes TOML config → calls `python3 template-processor.py` (extraction + theme generation + Jinja template rendering) |
| **Predefined** | `processPredefinedScheme(data, mode, wp)` | → `executePredefinedScheme()` → calls daemon IPC `ThemeGenerateFromPredefined` to expand 14→48 colours → writes expanded palette + TOML config → calls `python3 template-processor.py` for Jinja template rendering only |

### Terminal generation (Rust daemon)

Terminal configs are generated by the Rust daemon, not Python:

1. `handleTerminalThemesGenerate()` in `TemplateProcessor.qml` calls `CrawlService.sendRequest("ThemeGenerateTerminal", ...)`
2. The daemon calls `write_terminal_themes()` from `crawl-theme::apply::terminal`
3. Config files are written directly to disk by the daemon
4. Post-hooks are run from the callback for hot-reload

Supported terminals: foot, ghostty, kitty, alacritty, wezterm.
